## Summary

Implement both LFOs per ADR §4 and the parameter spec.

Note that vxn-1 already _has_ an implementation of two LFOs along the desired lines.

- **LFO1**: free-running OR BPM-sync, shared phase across all voices,
  evaluated once per control block (control rate, not sample rate).
- **LFO2**: per-voice (and per stacked instance), retriggered on note-on
  unless in Free mode, with delay (silence before LFO starts) and fade
  (linear ramp from 0 to full depth after delay completes).

Both share the same 6-waveform shape set: Sine, Tri, Saw+, Saw−, Pulse, S&H.

## Acceptance criteria

- [ ] `Lfo` struct: phase, rate (Hz), shape, optional delay/fade state.
- [ ] LFO1 lives at the patch level: one `Lfo` instance, evaluated per
      control block, written to a `mod_block.lfo1` field consumed by the
      matrix.
- [ ] LFO2 lives at the voice level: one `Lfo` per voice (and per stacked
      instance — 16 × 8 = 128 in the worst case). Evaluated per-block in
      the SoA voice batch.
- [ ] Waveforms: all 6 implemented per spec. Sine uses the project's
      Bhaskara+Moser approximation. S&H uses a sample-and-hold RNG seeded
      per LFO instance.
- [ ] BPM sync (LFO1 only): when `lfo1_sync` is on, snap `lfo1_rate` to the
      subdivision table (reuse VXN1's `vxn_app::sync::subdivisions`). Phase
      resets on transport restart.
- [ ] Delay + fade (LFO2 only): tracked in samples since note-on.
      0 .. `delay`: output 0. `delay` .. `delay + fade`: output ramps from
      0 to full. Past that: full depth. Free mode bypasses delay/fade —
      LFO2 runs from boot.
- [ ] KeySync (LFO2): when on, phase resets to 0 at each note-on. Free:
      phase continues across notes (still per-voice instance, so each
      voice has its own continuous phase).
- [ ] Bench: `lfo_block_eval` (cost of evaluating 128 LFO2s + 1 LFO1 per
      block).

## Notes

LFO depth itself is not part of the LFO struct — depth lives in mod matrix
slots routing FROM the LFO. The `lfo1_depth` and (implicit) LFO2 depth knob
on the panel are macro knobs that scale all matrix slots sourcing from that
LFO (a single multiplier applied at source eval time).

S&H phase is a counter that triggers a new random value at each cycle
boundary; the held value is the LFO's "output." Don't use the regular phase
accumulator to derive a triangle/saw and then quantise — that's wrong.

LFO2 in a stack of 8 voices, each with its own `voice_rand`-perturbed phase
(via matrix), is what produces the supersaw shimmer. Specifically: route
`voice_rand → lfo2_phase` with depth 1.0 in a default factory patch.
