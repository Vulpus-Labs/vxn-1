## Summary

Tenth ticket of [E006](../../epics/open/E006-review-remediation.md).
The review found zero CI for VXN-2: all three workflows under
`/.github/workflows/` (`build-windows.yml`, `release.yml`, `docs.yml`)
target vxn-1 only. Five closed epics, 60+ tickets, an extensive test
suite — never run automatically.

## Design

- New workflow `vxn2-test.yml` (or a job added to an existing
  workflow): `cargo test -p vxn2-dsp -p vxn2-engine -p vxn2-app
  -p vxn2-clap -p vxn2-ui-web` plus `cargo build -p vxn2-xtask`.
- Runner: `macos-latest` is the primary target (xtask bundle is
  macOS-only; `editor_smoke.rs` is macOS-gated). Add `ubuntu-latest`
  for the headless crates (dsp/engine/app) if cheap — they're
  platform-free and Linux runners are faster/cheaper.
- Path filter: trigger on `vxn-2/**` and `crates/vxn-core-*/**`
  changes (shared-core edits must run vxn-2 tests too — ticket 0067
  is exactly such a change).
- Cache cargo registry + target dir keyed on `Cargo.lock`.
- Benches: compile-check only (`cargo bench --no-run -p
  vxn2-osc-bench`) — running criterion in CI is noise, but bit-rotted
  bench code should fail the build.

## Acceptance criteria

- [ ] Workflow green on a clean push to main.
- [ ] A deliberately broken test on a branch fails the workflow
  (verify once, then revert).
- [ ] Shared-core path filter verified: a `vxn-core-app` touch
  triggers the vxn-2 job.
- [ ] `cargo bench --no-run` included.
- [ ] Total runtime reasonable (< ~10 min with warm cache).

## Notes

`editor_smoke.rs` spins the controller/IPC layer without instantiating
a real WebView, so it should pass headless on the macOS runner — but
verify; if it needs a window server, gate it behind an env var and
note the gap.
