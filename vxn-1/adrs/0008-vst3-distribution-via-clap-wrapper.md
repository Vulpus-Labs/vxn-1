# ADR 0008 — VST3 distribution via clap-wrapper

- **Status:** Accepted
- **Date:** 2026-06-08
- **Scope:** Adds a VST3 shipping artifact alongside the existing CLAP build.
  Selects free-audio/clap-wrapper as the bridge, single-binary bundled
  packaging, and macOS + Windows as the first-cut target matrix. Does not
  change the engine, the parameter model, the GUI, or the CLAP shell.

## Context

VXN1 ships today as a CLAP plugin only (ADR 0001 §1). The original rationale
for CLAP-only — recorded in ADR 0001 §1 — was licensing: `nih-plug`'s VST3
export pulled in GPLv3 via the Steinberg VST3 SDK's dual-licensing model,
and the project requires permissive (MIT / Apache-2.0) distribution. CLAP
via `clack` avoided that contamination.

Two things have changed.

**1. The VST3 SDK is now MIT.** Steinberg released VST 3.8 on
2025-10-29 under the MIT license, dropping the previous GPLv3-or-proprietary
dual model. Shipping a VST3 binary no longer requires a Steinberg agreement
and no longer creates a copyleft obligation. The constraint that pinned us
to CLAP-only is dissolved.

**2. Most DAW market share is still VST3 / AU.** Reaper, Bitwig, Live and a
few others load CLAP; the majority of Windows hosts (Cubase, Studio One, FL,
older Live), and Logic on macOS, do not. Shipping CLAP-only leaves a real
distribution gap that hurts adoption without protecting anything.

The remaining question is *how* to add VST3 without rewriting the plugin.
The two viable Rust paths are:

- **`nih-plug` rewrite** — port the shell to `nih-plug`, get VST3 + CLAP +
  standalone for free, lose the hand-rolled `clack` shell, the `vxn-app`
  controller integration ([[vxn1-architecture]], ADR 0007), and the HTML
  webview wiring. High blast radius, low new behavior.

- **`clap-wrapper`** — free-audio/clap-wrapper (MIT) wraps an existing CLAP
  plugin as VST3 / AUv2 / AUv3 / standalone. The CLAP remains the single
  source of truth; the wrapper translates host-side calls into CLAP-side
  calls at runtime. Used in production by Surge XT, Bitwig's own
  conversions, and most of the Free Audio cohort.

clap-wrapper is the right fit: it preserves every existing decision in ADRs
0001 / 0007, ships VST3 without owning a second parameter model, and lets us
add AU later from the same source.

## Decision

### 1. Adopt clap-wrapper as the VST3 bridge

VXN1 keeps its hand-rolled CLAP shell (`vxn-clap`, `clack`) as the canonical
plugin. VST3 is produced by feeding that CLAP through `clap-wrapper`. No
parameter, state, GUI, or processing code is duplicated; the wrapper
translates VST3-side calls into CLAP-side calls one-for-one.

Vendored as git submodules under `vendor/`:

- `vendor/clap-wrapper` — pinned to a release tag.
- `vendor/vst3sdk` — pinned to 3.8.x (MIT).

clap-wrapper's CMake build is invoked from `xtask`; nothing enters the Cargo
graph. The Rust workspace remains unaware of the wrapper.

### 2. Single-binary bundled packaging

The wrapper supports two packaging modes:

- **External CLAP** — the `.vst3` bundle contains a small shim that
  `dlopen`s a neighbouring `.clap` at runtime.
- **Bundled** — the wrapper statically links the CLAP into the `.vst3` so
  the VST3 file is self-contained.

We choose **bundled**. Reasons:

- No install-order coupling: the user's `.vst3` does not break if the
  matching `.clap` is uninstalled, moved or version-skewed.
- DAW validators (especially VST3 sandboxes on Windows) prefer
  self-contained bundles.
- A future AU build will need bundling anyway (Audio Unit validation is
  hostile to external dylibs).

The cost is a small build-side change: `vxn-clap` gains a `staticlib`
crate-type alongside its existing `cdylib`, so the same source produces both
the CLAP cdylib and a static archive the wrapper can link. The CLAP entry
symbol is generated by `clack`'s macro from either crate-type.

### 3. Target matrix — first cut

- **macOS VST3** (universal: arm64 + x86_64)
- **Windows VST3** (x86_64)

These two cover essentially every host that does not already load CLAP.
**AUv2** is intentionally out of scope for the first cut; the wrapper
supports it and we will add it in a follow-up ADR once the VST3 path is
stable. Linux VST3 is also deferred (low user base, simple to add later
since it's the same CMake invocation).

### 4. Build integration via `xtask`

`cargo xtask bundle` learns a `--format` flag accepting one or more of
`clap`, `vst3`. The CLAP path is unchanged. The VST3 path:

1. Builds `vxn-clap` as a staticlib (per architecture for `--universal`).
2. Invokes CMake on `vxn-1/wrapper/CMakeLists.txt`, passing the static
   archive path(s), the VST3 SDK path and the output name.
3. Copies the resulting `VXN1.vst3` bundle into `target/bundled/`.
4. `--install` copies to the platform's VST3 directory
   (`~/Library/Audio/Plug-Ins/VST3/` on macOS,
   `%LOCALAPPDATA%\Programs\Common\VST3\` on Windows).

CMake ≥ 3.21 is now a build prerequisite when building VST3. README is
updated; CI installs it on every runner.

### 5. Parameter identity and rename policy

clap-wrapper derives VST3 parameter IDs deterministically from CLAP IDs by
hashing. Renaming a CLAP parameter ID therefore breaks automation in saved
VST3 projects, even though it would not break CLAP automation
([[vxn1-id-stability-dropped]] — CLAP-side ID stability was abandoned in
favor of a clean param table).

Going forward, we adopt a soft rename policy *for VST3 ship only*:

- Adding parameters is free.
- Removing parameters is free (host shows "missing" automation in old
  projects; this is the DAW's job).
- Renaming a parameter's CLAP ID is a breaking change for VST3
  automation. Avoid it in releases; if unavoidable, note it in the
  release notes.

This is weaker than the original CLAP-id-stability discipline; it only
binds when the rename would be silent under CLAP.

### 6. Licensing — record the shift

VST 3.8 (October 2025) moved the VST3 SDK from dual GPLv3/Steinberg-proprietary
to MIT. clap-wrapper is MIT. clack is MIT-or-Apache-2.0. The entire shipping
chain is now permissive; the original ADR 0001 §1 rationale ("CLAP-only to
avoid GPLv3") no longer applies. This ADR does not supersede ADR 0001 §1
(CLAP remains the canonical format and the single source of truth); it
records the change in the environment that made the original constraint
moot.

## Consequences

**Positive.**

- VST3 shipping with no engine, parameter, GUI or state code duplicated.
- One source of truth for parameter and state behavior — bug fixes flow
  to both formats automatically.
- AUv2 path opens later via the same wrapper, no further architectural
  decision needed.
- License stack stays permissive.

**Negative.**

- New build prerequisite (CMake) on every dev machine and CI runner that
  wants VST3 output.
- Wrapper introduces a thin runtime translation layer (per-event,
  per-parameter). Expected sub-1% RT overhead but must be measured.
- Adds two git submodules (~80 MB) and a Windows CI runner.
- Soft rename policy for parameter IDs re-introduces a discipline we
  had dropped, narrower than before but real.

**Risks / open questions.**

- clap-wrapper's pinned VST3 SDK submodule may lag 3.8 at adoption time;
  override via CMake var to our own submodule if so.
- Webview lifecycle under Cubase historically aggressive
  (`IPlugView::attached` / `removed`); validate explicitly.
- Static-link path for `vxn-clap`: confirm `clack`'s entry-symbol macro
  exports cleanly from a `staticlib` crate-type (expected to, smoke-build
  before committing the wrapper CMake).

## Out of scope

- AUv2 / AUv3 (follow-up ADR).
- Standalone (`clap-wrapper` supports it; no demand yet).
- Linux VST3 (trivial follow-up).
- VST3 GUI features beyond what CLAP `gui` extension exposes — the
  wrapper translates verbatim, no new affordances.
